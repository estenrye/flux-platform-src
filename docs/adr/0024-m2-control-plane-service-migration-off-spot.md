# 24. M2: Control Plane Service Migration off Rackspace Spot

Date: 2026-07-13

## Status

Accepted — execution in progress. Steps 0-11 of the [design's execution
sequence](../superpowers/specs/2026-07-13-m2-migration-design.md#6-execution-sequence)
are complete as of 2026-07-21 (design, PKI ceremony, step-ca, cert-manager
stack, DNS cutover, Crossplane install, new delegated zone, delegated-zone
state migration, ESO/public DNS, soak gate, restore drill). The 7-day soak
runs through ~2026-07-28; steps 12 (go/no-go review), 13 (decommission),
and closing this ADR out as fully executed remain. This ADR will be
updated (not superseded) when decommission completes.

Finalizes the *placement* migration [ADR-20](0020-control-plane-on-talos-on-kvm.md)
declared but deferred ("Control plane services migrate from Spot in M2").
Amends [ADR-15](0015-secret-and-certificate-rotation-strategy.md) (SPIFFE
CA rotation section — see that ADR's 2026-07-21 amendment) and corrects a
stale line in ADR-20 (same date).

## Context

The fleet control plane — step-ca, the cert-manager/SPIFFE stack,
Crossplane, ESO, and public external-dns — ran on a Rackspace Spot
cloudspace with poor observed availability (~25% over the M0 audit's
sample window) and spend that never converges to zero as long as
anything depends on it. ADR-20 (M1) stood up `controlplane` (Talos on
KVM) as the replacement host but deferred service migration to M2.

The M0 baseline audit ([2026-07-11-m0-baseline-audit-design.md](../superpowers/specs/2026-07-11-m0-baseline-audit-design.md))
invalidated three assumptions the original plan (`fable-5-arch-plan.md`
§M2) carried into this milestone:

- The live SPIFFE trust domain was `cluster.local`, not a
  cluster-specific value (ADR-16 violation) — every Roles Anywhere trust
  anchor and ABAC condition would need to re-enroll regardless of
  whether the root identity changed, because the drift already broke
  the property ADR-16 exists to guarantee.
- `step-ca-db` had no backup configured and ran degraded (1/3 instances)
  — there was nothing to restore from a backup that didn't exist.
- The SOPS key-exposure incident (PR #66/#68) meant every credential in
  the `crossplane` 1Password vault had to be treated as compromised
  before anything touching it could be trusted for the new cluster.

Full design, decision rationale, and the five plan amendments (A1-A5)
this invalidation produced: [2026-07-13-m2-migration-design.md](../superpowers/specs/2026-07-13-m2-migration-design.md)
§1-2.

## Decision

Migrate each service to `controlplane` with Spot staying authoritative
until that service's cutover is verified (parallel run), then delete the
Spot cloudspace once a 7-day soak passes clean. The plan amendments:

- **A1/A5 — fresh PKI, not root preservation.** ADR-20 assumed the step-ca
  root would move unchanged; the drift finding made that not just
  unnecessary but actively worse (re-enrollment was mandatory either
  way, so keeping the old root would just carry forward a root that
  auto-rotated every 90 days by cert-manager design). New hierarchy: a
  **stable offline root** (`ryezone-labs Root CA`, 10y, ECDSA P-256,
  SOPS-only, private key never applied to any cluster) plus a **1-year
  `controlplane` intermediate** (`maxPathLen=1`, so step-ca's x5c
  provisioner can still issue workload-cluster intermediates from M4).
  Root fingerprint `6cdc50de…`, generated by a human running
  `.bin/generate-controlplane-pki.sh` so the root key never touches an
  agent session.
- **A2 — fresh `step-ca-db`.** No barman backup existed to restore;
  nothing state-critical to preserve. Backups are correct from day one:
  nightly `pg_dump` CronJob to TrueNAS NFS (interim, until barman
  targets Garage in M3), exercised by a restore drill (step 11,
  [docs/runbooks/step-ca-db-restore.md](../runbooks/step-ca-db-restore.md)).
- **A3 — `ca.rye.ninja` is a new name, not a repoint.** Repointing
  `ca.crossplane.rye.ninja` would serve a different root behind a
  fingerprint clients already pinned. `ca.crossplane.rye.ninja` keeps
  pointing at Spot until decommission, then dies with it.
- **A4 — old rendered repo archived at decommission**, not migrated;
  per-cluster rendered repos are the norm going forward.
- Consequence stated plainly in the design: because nothing state-critical
  moves for step-ca (fresh root, fresh DB), **the only true state
  migration in M2 is the Crossplane managed-resource set for the
  `crossplane.rye.ninja` delegated zone** — ten resources, executed at
  step 8 (2026-07-21). Orphan-equivalent `managementPolicies` → pause
  Spot Crossplane → export → import by external-name → observe-not-create
  → restore full management, verified independently against the AWS API
  (not just Crossplane's own status) — zero cloud-side recreation. The
  generalized pattern is now [docs/runbooks/crossplane-state-migration.md](../runbooks/crossplane-state-migration.md),
  reusable for M6+ cloud-substrate migrations, run here with a bounded
  blast radius (worst case: DNS breaks for a zone being retired anyway).
- Roles Anywhere re-enrollment (mandatory, not optional, per the drift
  finding): new trust anchor/profiles/ABAC pinned to
  `spiffe://controlplane.rye.ninja/...`, bootstrapped via a temporary
  static AWS credential before Crossplane's own SVID-derived credentials
  exist (chicken-and-egg: the new root isn't trusted by AWS until
  Crossplane, running on the new root, creates the trust anchor for it).

## Consequences

- `controlplane` is now the sole issuer of `ca.rye.ninja` certificates and
  the sole reconciler of the `crossplane.rye.ninja` delegated zone for
  the remainder of the soak. Spot's Crossplane is scaled to zero; its
  copies of the delegated-zone resources still exist (Orphan-equivalent
  policy protects them from deletion) and are removed only at
  decommission (step 13), when the claim is deleted **from
  `controlplane`** so Crossplane garbage-collects zone, records, and IAM
  cleanly.
- Two independently-trusted AWS Roles Anywhere trust anchors coexist
  through the soak (old root/`cluster.local` ABAC, new
  root/`controlplane.rye.ninja` ABAC) — accepted per the design's risk
  R5/R7 handling; the old one is deleted at decommission.
- The pinned-fingerprint-goes-stale failure mode the M0 audit flagged
  (90-day auto-rotating "root") is gone: the new root is stable for 10
  years by design, rotating only by deliberate ceremony. See ADR-15's
  2026-07-21 amendment for the revised rotation cadence.
- `ca.crossplane.rye.ninja` and the `crossplane.rye.ninja` zone are
  transitional — both retire at decommission (§5 of the design). Anything
  still resolving them after that point is a bug, not a slow migration.
- Two operational gotchas surfaced during execution, generalized into
  memory/runbooks rather than left as tribal knowledge: `deletionPolicy`
  doesn't exist on this Crossplane version (replaced by
  `managementPolicies`; see the state-migration runbook), and any new
  `cnpg.io/cluster` name on `controlplane` starts with zero traffic under
  the namespace's default-deny NetworkPolicy posture (see the restore
  runbook).
